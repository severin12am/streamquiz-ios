## 1. Overview

**WhoSmarter** is a real-time, multiplayer live quiz show for up to **6 players**. Each participant joins from their own browser, optionally shares their camera and microphone, and competes simultaneously on AI-generated questions. The experience is designed to feel like a TV quiz show: countdown timers, live scoreboards, camera tiles, and a winner screen at the end.

### Core concept

| Aspect | Detail |
|--------|--------|
| Players | 1 host (slot 0) + up to 5 guests (slots 1–5) |
| Questions | Generated by AI from a topic the host chooses |
| Sync | Supabase Realtime (Postgres changes + Broadcast channels) |
| Video | Peer-to-peer WebRTC mesh (optional cameras; mics always on) |
| Auth | None — link-based, anonymous access |
| Hosting | Next.js on Netlify (or any Node host) |

The app evolved from a **2-player buzzer quiz** into a **6-player simultaneous-answer** format. Legacy columns remain in the `games` table for backward compatibility but are **no longer used** by the application code.

## 2. Features

### Game creation (host)

- Choose **topic**, **difficulty** (easy / medium / hard), and **question count** (3–10)
- Toggle **multiple choice** vs **open-ended (voice/text)** answers
- Toggle **cameras** on or off (default: **on** — mics always work when off)
- Choose **game mode**: **First answer** (default) or **Think race** — see §6
- Choose **language** — English, Russian, Spanish, Arabic, French, German, or Japanese — affects UI and AI-generated questions
- QR code + shareable link for inviting players
- Session memory avoids repeating recent questions on the same topic

### Lobby

- Live player list with seat numbers (1–6)
- Host sees invite link + QR code
- Host can **START QUIZ** once at least one guest has joined
- Guests wait for the host to start

### During play

- Responsive **camera grid** (all players' streams when cameras enabled)
- Central **question panel** with circular countdown timer
- Live **leaderboard** (ScoreBoard)
- **Multiple choice**: everyone picks A/B/C/D within a timed window
- **Voice mode**: everyone speaks (or types) their answer, then locks in with **Done**
- Independent scoring — every correct player gets +1 per round
- Per-player ✓/✗ badges on camera tiles and in the result reveal

### End of game

- Ranked **winner screen** (handles ties)
- **Rematch voting** — host + at least one other player triggers a new round with fresh AI questions

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            SUPABASE                                     │
│  ┌─────────────────┐  ┌─────────────────┐  ┌────────────────────────┐ │
│  │  games table    │  │  players table  │  │  webrtc:{gameId} chan. │ │
│  │  (shared state) │  │  (per-player)   │  │  (SDP + ICE signals)   │ │
│  └─────────────────┘  └─────────────────┘  └────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────────┘
          ▲
          │  Server-only API calls
          │
┌─────────────────────────────────────────────────────────────────────────┐
│                     NEXT.JS SERVER (API Routes)                         │
│  POST /api/generate-questions  →  OpenRouter (gpt-4o / gpt-4o-mini)    │
│  POST /api/check-answer        →  OpenRouter (gpt-4o-mini judge)       │
│  GET  /api/ice-servers         →  STUN/TURN credentials (Metered, etc.) │
└─────────────────────────────────────────────────────────────────────────┘
```

### Separation of concerns

| State type | Where it lives | Who writes |
|------------|----------------|------------|
| Shared (question index, phase, deadline) | `games` row | Any client (guarded updates) |
| Per-player (pick, transcript, score, done) | `players` rows | Each client writes **only its own row** |
| WebRTC signaling | Supabase Broadcast channel | Peers exchange SDP/ICE |
| Media streams | Browser memory (P2P) | Never uploaded to server |

## 5. User Journey

### URLs

| URL | Role | Behavior |
|-----|------|----------|
| `/` | Host (creator) | Create game form |
| `/game/[id]?role=host` | Host | Takes slot 0; can start game and drive rematch |
| `/game/[id]` | Guest | Takes next free slot (1–5) |

The `role` query param is read server-side in `app/game/[id]/page.tsx` and passed to `GameScreen`. Only `?role=host` makes someone the host.

### Screen flow (GameScreen)

```
Loading → JoinScreen (if no seat) → Lobby (status=waiting)
    → Playing (camera grid + question panel) → WinnerScreen overlay (phase=ended)
```

Every participant runs the **same** `GameScreen` component. Host-only actions are gated by checking `me.role === 'host'` in hooks and UI.

### Identity persistence

- **`client_id`**: UUID stored in `localStorage` (`whosmarter-client-id`). Survives page reloads so a player re-attaches to the same seat.
- **Display name**: Saved in `localStorage` (`whosmarter-player-name`) and prefilled on JoinScreen.

## 6. Game Modes & Answer Types

### Game mode (`game_mode`)

UI labels and DB values:

| UI label | DB value | Behavior |
|----------|----------|----------|
| **Every answer counts** (default) | `regular` | Cooperative. Answer immediately; **everyone who answers correctly scores +1**. MC picks can be changed until the timer ends; voice answers can be edited until Done. No thinking phase, no timer shrink. |
| **Only first answer counts** | `hardcore` | Competitive. Answer immediately, but the answer **locks the instant you submit** (one-shot voice; first-tap MC). **Only the single earliest correct answer scores +1** (ranked by `answered_at` on the synced server clock). No thinking phase, no timer shrink. |

**Copy shown in create UI:**

- **Every answer counts:** “Answer right away and change your pick until the timer ends — everyone who answers correctly scores.”
- **Only first answer counts:** “Your answer locks the instant you submit. Only the first correct answer scores.”

Set at game creation in `CreateGame.tsx` (web) or under **Adjust** on the iOS home screen. Stored in `games.game_mode`.

Both modes use a **fixed 20s** answer timer that never shrinks, and end early only when **every** player has answered.

### `answered_at` (winner selection)

Each player row carries `answered_at` — an ISO timestamp stamped from the synced server clock (`serverNow()`) at the moment the player **first** commits (first MC pick, or voice Done). Reset to `null` each round. Used to pick the single winner in `hardcore`. In `regular` it is stored but not used for scoring.

### Legacy modes (read-only compatibility)

Old DB rows may still carry `think` / `classic`. The client keeps handling them:

| DB value | Behavior |
|----------|----------|
| `classic` | Immediate answer; first-answer timer shrink to 4s. |
| `think` | 5s locked `thinking` phase, then answer; first-answer timer shrink to 4s. |

`shrinksOnFirstAnswer(mode)` returns `true` only for `think` / `classic`. When **any** player answers in those modes, `useGameState.ts` shrinks `phase_deadline` so at most **4 seconds** remain (`FIRST_ANSWER_GRACE_SECONDS`). `regular` / `hardcore` never shrink.

### Answer type (`mc_mode`)

| Mode | Value | UI | Judging |
|------|-------|-----|---------|
| **Multiple choice** | `true` | A/B/C/D grid | Local string comparison against `correct_answer` |
| **Open-ended (voice)** | `false` | Speech recognition + text fallback + Done button | Local fuzzy match, then AI judge |

### Cameras (`cameras_enabled`)

| Setting | Default | Effect |
|---------|---------|--------|
| `true` | ✓ (default) | 720p camera + mic sent to all peers |
| `false` | | Mic only — lighter mesh, less bandwidth |

### Default create form

| Field | Default |
|-------|---------|
| `difficulty` | `medium` |
| `num_questions` | `5` |
| `mc_mode` | `true` (multiple choice) |
| `game_mode` | `regular` (**Every answer counts**) |
| `cameras_enabled` | `true` |
| `locale` | `en` (or device/browser language) |

## 7. State Machine & Round Flow

### Game status (`games.status`)

| Status | Meaning |
|--------|---------|
| `waiting` | Lobby — not started |
| `ready` | (reserved / legacy) |
| `playing` | Match in progress |
| `ended` | All questions finished |

### Game phase (`games.phase`)

| Phase | Active when | Duration |
|-------|-------------|----------|
| `waiting` | Lobby | Until host starts |
| `thinking` | Legacy `think` mode only | 5s (`THINK_TIME_SECONDS`) |
| `question` | MC mode — picking answers | 20s (`QUESTION_TIME_SECONDS`) |
| `answering` | Voice mode — speaking | 20s (`VOICE_ANSWER_SECONDS`) |
| `checking` | AI judging voice answers | Up to 15s safety timeout |
| `result` | Show outcomes | 5s (`RESULT_TIME_SECONDS`) |
| `ended` | Game over | — |

Legacy phases `buzzing` and `judging` exist in the DB schema but are **unused** in the current app.

### Round flow diagrams

**Multiple choice (`regular` / `hardcore`):**

```
question (20s or early if all picked) → result (5s) → next question
```

**Voice (`regular` / `hardcore`):**

```
answering (20s or early if all Done) → checking → result (5s) → next question
```

`regular` and `hardcore` go straight to `question` / `answering` (no `thinking`). Legacy `think` prepends a 5s locked `thinking` phase.

### Phase advancement (deadline-driven)

All timed phases store an ISO timestamp in `games.phase_deadline`. **Every client** runs a local ticker (100ms interval) in `useGameState.ts`:

1. Recompute countdown from `phase_deadline` using **server-synced clock**
2. If deadline expired → attempt guarded phase transition via `updateGameIfPhase()`
3. Postgres `UPDATE ... WHERE phase = expectedPhase` ensures **only one client wins** the race

This design means the game continues even if the **host disconnects** — any participant can drive transitions.

### Early advance

| Phase | Condition | Action |
|-------|-----------|--------|
| `question` | Every player has `mc_index !== null` | `resolveMcRound()` |
| `answering` | Every player has `done === true` | Transition to `checking`, run `runVoiceCheck()` |

### First-answer grace (deadline shrink — LEGACY only)

| Trigger | Condition | Action |
|---------|-----------|--------|
| Any player answers | `game_mode` is `think`/`classic`, `phase` is `question`/`answering`, and `hasAnswered(player)` becomes true for someone | If more than 4s remain on `phase_deadline`, shrink to **4s from now** via `updateGameIfDeadline()` |

Constants: `FIRST_ANSWER_GRACE_SECONDS = 4`. Gated by `shrinksOnFirstAnswer(mode)` — applies **only** to legacy `think` / `classic`. `regular` and `hardcore` never shrink the timer.

### Guarded updates

```typescript
// Only the first client whose update matches the current phase succeeds
updateGameIfPhase(gameId, 'question', { phase: 'result', ... })
```

Defined in `lib/supabase.ts`. This prevents double-scoring and duplicate transitions.

## 8. Scoring & Answer Judging

### Scoring rules (current)

- **+1 point** per correct answer per round
- Each player judged **independently** — multiple players can score on the same question
- No streaks, steals, or multipliers (removed in earlier versions; legacy DB columns remain)

### Multiple choice

1. Player calls `submitMCAnswer(index)` → writes `mc_index` on **their** `players` row
2. At round end, `resolveMcRound()` fetches authoritative player list
3. `isMcAnswerCorrect()` compares chosen option text to `correct_answer` (normalized)
4. Winners get `score + 1` and `correct: true`

### Voice / open-ended

1. During `answering`, speech updates `transcript` on the player's row live
2. Player clicks **Done** → sets `done: true` (optional final transcript)
3. When round ends, one client wins the `checking` guard and calls `runVoiceCheck()`
4. Each transcript is judged via `POST /api/check-answer`:
   - **Layer 1**: Local Unicode-aware fuzzy match against `correct_answer` and `accepted_answers`
   - **Layer 2**: AI judge (gpt-4o-mini) returns YES/NO
5. Scores updated per player; `games.answer_correct` set to `true` if **any** player was correct (drives shared UI)

## 9. Database Schema

### `games` table

**Setup fields** (immutable after creation):

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID | Primary key |
| `topic` | TEXT | Quiz subject |
| `difficulty` | TEXT | `easy`, `medium`, `hard` |
| `num_questions` | INT | 3–10 |
| `mc_mode` | BOOLEAN | MC vs voice |
| `cameras_enabled` | BOOLEAN | Request video streams |
| `game_mode` | TEXT | `regular` (Every answer counts) or `hardcore` (Only first answer counts); legacy `think` / `classic` still accepted |
| `questions` | JSONB | Array of `Question` objects |

**Live fields** (updated during play):

| Column | Type | Description |
|--------|------|-------------|
| `status` | TEXT | `waiting`, `playing`, `ended` |
| `current_question_index` | INT | 0-based |
| `phase` | TEXT | Current round phase |
| `phase_deadline` | TIMESTAMPTZ | Auto-advance timestamp |
| `answer_correct` | BOOLEAN | Any player correct this round |
| `last_points` | INT | Points on last correct (for animations) |

**Legacy 2-player columns** (still in schema, **not used by app**):

`buzz_player`, `buzz_time`, `host_score`, `player_score`, `host_transcript`, `player_transcript`, `host_correct`, `player_correct`, `host_done`, `player_done`, `host_mc_index`, `player_mc_index`, `is_steal`, `first_answerer`, `streak_host`, `streak_player`, `last_scorer`, `rematch_host`, `rematch_player`

### `players` table

One row per participant. Unique constraints: `(game_id, slot)` and `(game_id, client_id)`.

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID | Used as WebRTC peer id |
| `game_id` | UUID | FK → games |
| `client_id` | TEXT | Browser-stable id |
| `name` | TEXT | Display name |
| `role` | TEXT | `host` or `player` |
| `slot` | INT | 0 (host) .. 5 |
| `score` | INT | Running total |
| `mc_index` | INT | MC pick this round (0–3 or null) |
| `transcript` | TEXT | Voice answer this round |
| `correct` | BOOLEAN | Judged outcome this round |
| `done` | BOOLEAN | Voice lock-in this round |
| `rematch` | BOOLEAN | Rematch vote |
| `answered_at` | TIMESTAMPTZ | Server-clock time of first commit this round (winner selection in `hardcore`); reset each round |

### Row Level Security

Both tables use **fully open** anonymous policies (`USING (true) WITH CHECK (true)`). Appropriate for private link-based games; replace with auth policies for public deployment.

### Realtime publication

Both `games` and `players` are added to `supabase_realtime` publication for live sync.

## 10. Real-Time Synchronization

### Subscriptions (`lib/supabase.ts`)

| Channel | Event | Purpose |
|---------|-------|---------|
| `game:{gameId}` | `UPDATE` on `games` | Phase, question, deadline changes |
| `players:{gameId}` | `*` on `players` | Join/leave, picks, scores |

On player changes, clients **refetch the full player list** (simple, consistent for ≤6 rows).

### Polling fallback

If Realtime is blocked (firewall, region), `useGameState` polls every **2.5 seconds** as a backup.

### Server clock sync

Device clocks often drift by seconds. All deadline comparisons use `serverNow()`:

1. `syncServerClock()` HEAD-requests Supabase REST API
2. Reads HTTP `Date` header
3. Computes offset: `serverOffsetMs`
4. Re-syncs every 30 seconds

Without this, one device could race through questions while another lags.

### Initial load retries

Up to **5 attempts** with 1.2s delay if game fetch fails (unstable networks / geo-blocking).

## 11. WebRTC Camera Mesh

Implemented in `hooks/useMeshWebRTC.ts`.

### Topology

**Full mesh**: N players → N×(N−1)/2 peer connections. Each browser uploads its stream to every other player. Fine for ~6 participants; beyond that you'd want an SFU (media server).

### Signaling

- Supabase Realtime **Broadcast** channel: `webrtc:{gameId}`
- Messages typed as `WebRTCSignal`: `{ type, from, to, payload }`
- Routed by player `id` — each peer ignores messages not addressed to them

### Presence

Supabase **Presence** keyed by `myId` (player UUID):

- `join` / `leave` / `sync` → reconcile peer connections
- New peer online → `ensurePeer()` opens RTCPeerConnection
- Peer offline → `teardownPeer()` closes connection

### Perfect Negotiation

Politeness is deterministic: **lower player id = polite**. On offer collision, impolite side wins; polite side rolls back. Impolite side also triggers **ICE restart** on connection failure.

### ICE servers

Fetched from `GET /api/ice-servers` (server-side credentials). Fallback chain:

1. **Metered** (if `METERED_DOMAIN` + `METERED_API_KEY` set)
2. **ExpressTURN** or static creds (if `TURN_URLS` + `TURN_USERNAME` + `TURN_CREDENTIAL` set)
3. Public STUN + free TURN relay (limited reliability)

### Camera constraints

When `cameras_enabled`:

```typescript
video: { width: { ideal: 1280 }, height: { ideal: 720 }, facingMode: 'user', frameRate: { ideal: 30 } }
audio: true  // always
```

When cameras off: `video: false`, `audio: true`.

## 13. AI Integration

### Provider

The OpenAI SDK points to **OpenRouter** (`https://openrouter.ai/api/v1`). The env var is named `OPENAI_API_KEY` but accepts an OpenRouter key (`sk-or-v1-...`). To use native OpenAI, remove the `baseURL` in the API route files.

### Question generation prompts

`lib/quiz-prompts.ts` defines difficulty-specific guides:

- **Easy**: Pop-culture, widely known facts
- **Medium**: Enthusiast-level, specific works/events
- **Hard**: Obscure expert trivia

Prompts enforce: single unambiguous answers, JSON-only output, topic spread, no repeats from `previous_questions`.

### Question memory

`lib/question-history.ts` stores up to 24 recent question texts per topic in `sessionStorage`. Used on create and rematch to reduce AI repetition within a browser session.

### Answer judging philosophy

- **Lenient** for speech artifacts (filler words, STT errors)
- **Strict** about factual correctness
- AI temperature `0` for deterministic judging
- On API failure → defaults to **wrong** so the game continues

## 16. Hooks Reference

### `useGameState(gameId, clientId)`

**Returns:** game, players, me, loading, error, timers, and actions.

| Action | Who | Effect |
|--------|-----|--------|
| `join(name, asHost)` | Anyone | Claim or re-attach seat |
| `startGame()` | Host only | Reset scores, begin round 1 |
| `submitMCAnswer(i)` | Anyone in MC phase | Set pick (changeable in `regular`, locked on first tap in `hardcore`); stamps `answered_at` on first pick |
| `updateTranscript(text)` | Voice phase | Live transcript sync (until `done`) |
| `finishAnswer(text?)` | Voice phase | Mark `done`; stamps `answered_at` |
| `voteRematch()` | Anyone | Set `rematch: true` |
| `rematch(questions?)` | Host (triggered by votes) | Reset to lobby |

### Known limitations

- **No authentication** — anyone with the link can read/write game data
- **Mesh bandwidth** — scales O(N²); 6 players is the practical cap
- **AI cost** — each game + rematch calls OpenRouter; each voice round may call check-answer per player
- **Speech on Firefox** — users must type answers manually
- **Supabase geo-blocking** — some regions need VPN (error message shown)

## 24. Network Resilience

The app includes multiple layers for unreliable networks:

1. **Realtime + 2.5s polling fallback**
2. **Server clock sync** (prevents desync)
3. **Guarded DB updates** (prevents race corruption)
4. **Initial load retries** (5×)
5. **ICE restart** on WebRTC failure (impolite peer)
6. **TURN relay** for strict NAT / VPN
7. **Checking phase timeout** — if AI judge client vanishes, round still advances
8. **Think race mode** — locked `thinking` phase prevents fast-connection advantage on question reveal
9. **First-answer grace** — 4s deadline shrink when anyone answers (both modes)

## 26. Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| Blank page / setup banner | Missing `.env.local` | Copy example, add Supabase keys, restart dev server |
| "Couldn't load the game" | Bad link or Supabase blocked | Verify UUID; try VPN |
| Black remote video | NAT / no TURN | Configure Metered or ExpressTURN env vars |
| Questions out of sync | Clock skew (rare now) | Ensure `syncServerClock` runs; check device time |
| Voice not working | Firefox or no mic permission | Use Chrome; allow mic; or type answer |
| Game full | 6 seats taken | Wait for someone to leave or create new game |
| AI errors on create | Bad/missing API key | Check `OPENAI_API_KEY` in env |
| Russian answers always wrong | Was Unicode bug | Fixed in `check-answer` — ensure latest code |
| Rematch same questions | AI ignored avoid list | Check `question-history` + `previous_questions` payload |
