## 1. What It Is

**WhoSmarter** is a real-time, multiplayer live quiz show for up to **6 players**. Each participant joins from their own device — browser (Chrome, Safari, Edge) or native iOS app — optionally shares camera and microphone, and competes **simultaneously** on AI-generated questions about a topic the host chooses. The experience is designed to feel like a TV quiz show: countdown timers, live scoreboards, camera tiles with per-player ✓/✗ badges, and a ranked winner screen at the end.

This repository (`whosmarter-ios`) is the **native iOS client** built with **Expo 54 / React Native 0.81**. It plays in the **same Supabase games** as the companion **Next.js web app** deployed at `https://streamquiz.netlify.app`. Cross-platform play is a first-class feature: a host on iPhone can invite guests on Chrome; a web host can invite guests on iPhone — all participants share one game state, one phase clock, and one scoreboard.

| Aspect | Detail |
|--------|--------|
| Players | 1 host (slot 0) + up to 5 guests (slots 1–5) |
| Questions | Generated by AI (OpenRouter → GPT-4o / GPT-4o-mini) from a host-chosen topic |
| Sync | Supabase Realtime (Postgres row changes + Broadcast channels for WebRTC) |
| Video | Peer-to-peer WebRTC full mesh (optional cameras; mics always available) |
| Auth | None — link-based, anonymous access |
| Monetization (iOS) | **Joining is always free.** Creating quizzes is gated: 5-quiz free trial, then optional subscriptions via StoreKit/RevenueCat |
| AI & TURN secrets | Server-only on the deployed Next.js API — **never** bundled in the mobile app |

The app evolved from an earlier **2-player buzzer quiz** into a **6-player simultaneous-answer** format. Legacy database columns and unused UI (`BuzzButton`) remain for backward compatibility but are not part of the active flow.

## 2. Product Vision & Design Goals

### Feel like a live show

WhoSmarter is not a turn-based trivia app. Every player sees the same question at the same time, hears the same ticking clock, and watches scores update live on camera tiles. The lagoon-themed light UI (teal accents, soft greens, keycap-style buttons) mirrors the web app's CSS variables so the brand feels consistent across platforms.

### Fairness without a dedicated server

There is no central game server process. Instead, **every client** runs the same state machine and competes to advance phases via **guarded Postgres updates** — only one client wins each transition. A **server-synced clock** (`syncServerClock()`) prevents one fast device from racing ahead of others. In `regular` mode everyone who answers correctly scores; in `hardcore` mode only the earliest correct answer counts, ranked by `answered_at` stamped from the synced server clock.

### Optional video, always social

Cameras default **on** (720p ideal, dynamically scaled down as more peers join). Hosts can disable cameras for a mic-only, lighter mesh. Push-to-talk between rounds lets players talk without open-mic chaos. During voice-answer phases, the local player's mic is muted to peers (Speech recognition owns the microphone on iOS).

### Private-party first

No accounts, no login walls. Share a link or QR code. Appropriate for friends-and-family sessions; not hardened for public anonymous internet scale without additional RLS and rate limiting.

### Creator monetization (iOS)

On the native iOS app, **hosting** (creating new quizzes) is monetized: a **5-quiz lifetime free trial** per install, then optional **limited** (30/month) or **unlimited** subscriptions through Apple In-App Purchase. **Joining and playing remain free** for everyone, on every platform.

## 3. The Ecosystem (Web + iOS + Swift)

| Client | Stack | Role |
|--------|-------|------|

## 4. Core User Journey

### 4.1 App launch & Home screen

`HomeScreen.tsx` is the entry point (route `/` on web, stack root on iOS).

**Create a game (host):**

1. Enter a **topic** (required) — e.g. "90s movies", "World capitals".
2. Tap **Create Challenge** (or open **Adjust** first for fine-grained settings).
3. App **checks create quota** via `EntitlementsProvider.refresh()` — if free trial or monthly cap is exhausted, navigates to **Paywall** without calling AI (see §25).
4. App calls `POST /api/generate-questions` with topic, difficulty, count, MC vs voice, `game_mode`, locale, and `previous_questions` dedup list from session history.
5. App inserts a new row into Supabase `games` with returned questions, `status: 'waiting'`, `phase: 'waiting'`.
6. App calls `noteCreated()` to increment the local usage counter.
7. Navigates to **Game** with `asHost: true` → host claims slot 0.

A **quota pill** on Home shows remaining allowance (e.g. "3 free quizzes left" or "12 left this month" for `limited` subscribers). Tapping it opens the paywall. **Joining is never gated.**

**Join a game (guest):**

- Paste a share link or raw UUID into the join field on Home.
- Or arrive via **Universal Link** (`https://streamquiz.netlify.app/game/{uuid}`) or custom scheme (`whosmarter://game/{uuid}`; legacy `streamquiz://` also supported).
- Navigates to **Game** with `asHost: false` → next free slot 1–5.

**Adjust panel settings** (expandable on Home): difficulty, question count (3–10), MC vs voice, game mode (`regular` / `hardcore`), cameras on/off, language (via `LanguagePicker` on Home header area).

If env vars are missing, a setup banner blocks creation (`isConfigured()` checks all three `EXPO_PUBLIC_*` vars).

### 4.2 Join screen

If the player has no seat yet (`me === null`), `JoinScreen` prompts for a **display name**. Names persist in AsyncStorage (`whosmarter-player-name`) and prefill on return visits. A stable **`client_id`** UUID in AsyncStorage (`whosmarter-client-id`) lets the same device re-attach to the same seat after app restart or navigation away — important for flaky mobile networks.

Join logic (`joinGame` in `supabase.ts`):

- Slot **0** reserved for host when `asHost: true`.
- Slots **1–5** assigned to guests in order of availability.
- If all 6 seats taken → "game full" UI.
- Reattach: if `client_id` already has a row in this game, return that player.

### 4.3 Lobby

While `games.status === 'waiting'`:

- **Player list** with colored avatars (slot-based colors from `player-colors.ts`), host badge, "(you)" marker, empty seat placeholders up to `MAX_PLAYERS` (6).
- **Host only:** QR code (`react-native-qrcode-svg`) encoding `gameShareUrl(gameId)` — guests scan with iPhone Camera app or open in Safari/Chrome.
- **Copy link** → clipboard; Share sheet via `Share` API.
- **START QUIZ** enabled when `players.length >= 2` (host + at least one guest).
- Guests see a waiting message; no start button.

Sound: `useGameSounds` plays a **join** chime when a new player row appears.

### 4.4 During play

`GameScreen` orchestrates the playing UI once `status === 'playing'`.

**Layout (WhatsApp-style group call):**

- **Full-screen camera layer** — opponent video tiles fill the background (`CameraGrid` with `fill` mode). One opponent = full screen; 2+ opponents = 2-column grid rows.
- **Self PiP** — your own feed is a small picture-in-picture tile (bottom corner) when others are present; you mostly see opponents large.
- **Floating question overlay** — `QuestionPanel` with `dark` mode renders translucent chips over the video: top section (round label, question text, circular timer), bottom section (MC grid or voice UI / result rows).
- **Mic toggle** — push-to-talk hold or mute; respects mic policy (see §14).
- **Sound toggle** — mutes game SFX globally.

**Multiple choice (`mc_mode: true`):**

- Phase `question`, **20 seconds** (`QUESTION_TIME_SECONDS`).
- A/B/C/D grid (`MCOptions`); avatars appear on options as players pick.
- **`regular`:** change pick until timer ends; all correct players score +1.
- **`hardcore`:** first tap locks pick and stamps `answered_at`; only earliest correct answer scores.
- Early advance: when **every** player has `mc_index !== null`, any client calls `resolveMcRound()`.

**Voice / open-ended (`mc_mode: false`):**

- Phase `answering`, **20 seconds** (`VOICE_ANSWER_SECONDS`).
- Apple Speech (`@react-native-voice/voice`) transcribes live on physical iPhone; Simulator falls back to typed input.
- Transcript syncs to Supabase on the player's row, **throttled to 350ms** to limit writes.
- **Done** button locks answer (`done: true`, stamps `answered_at` on first commit).
- Early advance: when **every** player has `done === true` → phase `checking` → AI/local judge per player.
- Banner: "answering muted" — peers cannot hear you during voice phase (WebRTC mic off; Speech owns mic).

**Result phase:**

- **5 seconds** (`RESULT_TIME_SECONDS`) showing per-player ✓/✗ on camera tiles and in question panel result list.
- MC mode may keep the option grid visible with correct answer highlighted.

**Between questions:**

- Scores reset per-round fields (`mc_index`, `transcript`, `done`, `correct`, `answered_at`) via `resetPlayersForRound`.
- `current_question_index` increments; if past last question → `status: 'ended'`, `phase: 'ended'`.

### 4.5 End of game & rematch

`WinnerScreen` overlay when `phase === 'ended'` or `status === 'ended'`:

- Ranked list sorted by score (ties broken by lower slot number).
- Sole winner hero card with trophy emoji; tie shows multiple names; zero-score game shows handshake emoji.
- **Rematch** button sets `rematch: true` on the player's row.
- When **host** and **≥1 guest** have voted, host client auto-fetches fresh AI questions (merged dedup history) and calls `rematch(questions)` — returns to lobby with reset scores.

## 5. Game Settings & Modes (Deep Dive)

Set at creation in `CreateGame.tsx` (defaults below). Stored immutably on the `games` row after insert.

| Setting | DB field | Default | Options / behavior |
|---------|----------|---------|-------------------|
| Topic | `topic` | (required) | ≤200 chars; drives AI prompt |
| Difficulty | `difficulty` | `medium` | `easy` → gpt-4o-mini; `medium`/`hard` → gpt-4o |
| Question count | `num_questions` | `5` | Clamped 3–10 |
| Answer type | `mc_mode` | `true` | MC (A/B/C/D) vs voice/open-ended |
| Game mode | `game_mode` | `regular` | See below |
| Cameras | `cameras_enabled` | `true` | Video+audio vs mic-only mesh |
| Language | (client `locale`) | device default | 7 locales; affects UI + AI + speech |

### Game mode: `regular` vs `hardcore`

| UI label | DB value | Scoring | Answer locking |
|----------|----------|---------|----------------|
| **Every answer counts** | `regular` | All correct players +1 | MC: change until timer; voice: edit until Done |
| **Only first answer counts** | `hardcore` | Single earliest correct +1 | MC: lock on first tap; voice: lock on Done |

**Copy in create UI:**

- *Every answer counts:* "Answer right away and change your pick until the timer ends — everyone who answers correctly scores."
- *Only first answer counts:* "Your answer locks the instant you submit. Only the first correct answer scores."

`answered_at` (ISO timestamp from `serverNow()`) is written on first MC pick or voice Done. Reset each round. Used for hardcore winner selection.

### Legacy modes (read-only compatibility)

Old games may still have `think` or `classic` in `game_mode`:

| DB value | Behavior |
|----------|----------|
| `think` | 5s locked `thinking` phase, then answer; timer shrinks to 4s grace when anyone answers |
| `classic` | Immediate answer; same 4s first-answer grace shrink |

New games from this client use `regular` or `hardcore` with **fixed 20s timers** that never shrink. `shrinksOnFirstAnswer()` returns true only for legacy modes.

## 6. State Machine & Round Flow

### Game status (`games.status`)

| Status | Meaning |
|--------|---------|
| `waiting` | Lobby — not started |
| `ready` | Reserved / legacy |
| `playing` | Match in progress |
| `ended` | All questions finished |

### Game phase (`games.phase`)

| Phase | When | Duration |
|-------|------|----------|
| `waiting` | Lobby | Until host starts |
| `thinking` | Legacy `think` mode only | 5s |
| `question` | MC mode — picking | 20s (or early if all picked) |
| `answering` | Voice mode — speaking | 20s (or early if all Done) |
| `checking` | AI judging voice answers | Up to 15s safety timeout |
| `result` | Show outcomes | 5s |
| `ended` | Game over overlay | — |

Legacy phases `buzzing` and `judging` exist in schema but are **unused**.

### Round flow diagrams (current modes)

**Multiple choice (`regular` / `hardcore`):**

```
question (20s) → result (5s) → next question OR ended
         ↑ early if all mc_index set
```

**Voice (`regular` / `hardcore`):**

```
answering (20s) → checking → result (5s) → next question OR ended
         ↑ early if all done=true
```

### Phase advancement mechanics

1. Host `startGame()` sets `status: 'playing'`, resets scores, applies `roundStartPatch()` for question 0.
2. Every client runs a **100ms ticker** (`TICK_INTERVAL_MS`) comparing `phase_deadline` to `serverNow()`.
3. On expiry, client attempts **guarded update**: `updateGameIfPhase(gameId, expectedPhase, patch)`.
4. Postgres `UPDATE ... WHERE phase = expectedPhase` — **only one client wins** the race.
5. Winner runs scoring (`resolveMcRound` / `runVoiceCheck`) before advancing.

**Early advance triggers:**

| Phase | Condition | Action |
|-------|-----------|--------|
| `question` | All players `mc_index !== null` | `resolveMcRound()` |
| `answering` | All players `done === true` | → `checking`, then `runVoiceCheck()` |

**Legacy first-answer grace** (think/classic only): when any player answers, if >4s remain on deadline, shrink to 4s from now via `updateGameIfDeadline()`.

### Timing constants (must match web)

```typescript
QUESTION_TIME_SECONDS = 20
VOICE_ANSWER_SECONDS  = 20
RESULT_TIME_SECONDS   = 5
CHECK_TIMEOUT_SECONDS = 15
THINK_TIME_SECONDS    = 5      // legacy only
FIRST_ANSWER_GRACE_SECONDS = 4 // legacy only
POLL_INTERVAL_MS      = 2500
TICK_INTERVAL_MS      = 100
```

Exported from `useGameState.ts` and unit-tested in `parity.test.ts`.

## 7. Database & Shared State

### `games` table (summary)

**Setup (immutable after create):** `topic`, `difficulty`, `num_questions`, `mc_mode`, `cameras_enabled`, `game_mode`, `questions` (JSONB array).

**Live (updated during play):** `status`, `current_question_index`, `phase`, `phase_deadline`, `answer_correct`, `last_points`.

**Legacy 2-player columns** still in schema but unused by app code: `buzz_player`, `host_score`, streaks, steals, etc.

### `players` table (summary)

One row per participant. Unique: `(game_id, slot)` and `(game_id, client_id)`.

| Column | Purpose |
|--------|---------|
| `id` | UUID — also WebRTC peer id |
| `slot` | 0 = host, 1–5 = guests |
| `score` | Running total |
| `mc_index` | MC pick this round (0–3 or null) |
| `transcript` | Voice answer text |
| `correct` | Judged outcome this round |
| `done` | Voice lock-in |
| `rematch` | Rematch vote |
| `answered_at` | Server-clock first commit (hardcore winner) |

### Row Level Security

Both tables use **fully open** anonymous policies (`USING (true) WITH CHECK (true)`). Fine for private link games; tighten for public deployment.

Both tables are in `supabase_realtime` publication.

### Separation of concerns

| State type | Storage | Writer |
|------------|---------|--------|
| Shared round state | `games` row | Any client via guarded updates |
| Per-player picks/scores | `players` rows | Each client writes **only its own row** |
| WebRTC signaling | Broadcast `webrtc:{gameId}` | Peers (typed `WebRTCSignal`) |
| Media streams | In-memory P2P | Never uploaded |

The RN app is intentionally a **thin native client**. All AI and TURN secret handling stays on the Next.js deployment.

## 9. Real-Time Synchronization

### Realtime channels (`lib/supabase.ts`)

| Channel | Events | Purpose |
|---------|--------|---------|
| `game:{gameId}` | `UPDATE` on `games` | Phase, deadline, question index |
| `players:{gameId}` | `*` on `players` | Join/leave, picks, scores, rematch |

On any player change, clients **refetch the full player list** (simple and consistent for ≤6 rows).

### Polling fallback

If Realtime is blocked (firewall, region), `useGameState` polls every **2.5 seconds** (`POLL_INTERVAL_MS`).

### Server clock sync

Device clocks drift. All deadline math uses `serverNow()`:

1. `syncServerClock()` HEAD-requests Supabase REST API
2. Reads HTTP `Date` header
3. Sets `serverOffsetMs = serverTime - Date.now()`
4. Re-sync when `shouldResyncClock()` (>30s since last sync)

Without this, one device could expire phases seconds before another.

### Initial load resilience

Up to **5 attempts** (`MAX_INIT_ATTEMPTS`) with **1.2s** delay (`INIT_RETRY_DELAY_MS`) if game fetch fails — helps unstable networks and geo-blocked regions (user may need VPN).

## 10. WebRTC Camera Mesh

Implemented in `hooks/useMeshWebRTC.ts`. Ported from web with iOS-specific lifecycle hooks.

### Topology

**Full mesh:** N players → N×(N−1)/2 peer connections. Each device uploads its stream to every other player. Practical cap ~6; beyond that you'd want an SFU.

### Signaling

- Supabase Realtime **Broadcast** channel: `webrtc:{gameId}`
- Messages: `{ type: 'offer' | 'answer' | 'ice-candidate', from, to, payload }`
- Routed by player `id` — ignore messages not addressed to self

### Presence

Supabase **Presence** keyed by `myId`:

- `join` / `leave` / `sync` → `reconcile()` peer map
- New peer online → `ensurePeer()` opens `RTCPeerConnection`
- Peer offline → `teardownPeer()` after **8s grace** (`PRESENCE_GRACE_MS`) to survive flicker

### Perfect Negotiation

Politeness is deterministic: **lower player UUID string = polite**. On offer collision, impolite side wins; polite side rolls back. Impolite side triggers **ICE restart** on connection failure.

### ICE servers

Fetched from `GET /api/ice-servers` before any `RTCPeerConnection` is created (`ensureIceServers()` — critical: empty ICE list = host candidates only = black video across NAT).

Fallback chain on server: Metered → ExpressTURN/static → public STUN. Client merges Google STUN extras in `api/client.ts`.

### Dynamic video quality

Resolution/bitrate tiers scale down as peer count grows (`videoTierForPeerCount`):

| Peers | Resolution | FPS | Max bitrate |
|-------|------------|-----|-------------|
| ≤2 | 1280×720 | 30 | 1200 kbps |
| ≤3 | 960×540 | 24 | 900 kbps |
| 4+ | 640×480 | 20 | 600 kbps |

When `cameras_enabled: false`: `video: false`, `audio: true`.

### iOS lifecycle recovery

- **AppState** foreground → restart capture, re-announce presence, reconcile, ICE-restart unhealthy peers
- **NetInfo** regain/switch → same recovery path
- **3s reconcile timer** (`RECONCILE_INTERVAL_MS`) — idempotent `ensurePeer()` self-corrects missed presence events

Tracks start **disabled**; `GameScreen` calls `setMicEnabled` per mic policy.

## 11. Scoring & Answer Judging

### Rules

- **+1 point** per correct answer per round
- **`regular`:** multiple players can score on the same question
- **`hardcore`:** only one winner — earliest `answered_at` among correct players
- No streaks, steals, or multipliers (removed in earlier versions)

### Multiple choice flow

1. `submitMCAnswer(index)` → writes `mc_index` (+ `answered_at` on first pick in hardcore)
2. Round end: `resolveMcRound()` fetches authoritative player list
3. `isMcAnswerCorrect()` compares chosen option text to `correct_answer` (normalized via `mc-utils.ts`)
4. Winners: `score + 1`, `correct: true`; sets `games.answer_correct` if any player correct

### Voice flow

1. Live `transcript` updates during `answering` (throttled)
2. `finishAnswer()` → `done: true`
3. Phase `checking`: one client wins guard, runs `runVoiceCheck()`
4. Per player: `checkAnswer()` → `POST /api/check-answer`
   - **Layer 1:** Local Unicode-aware fuzzy match (`\p{L}\p{N}`) against `correct_answer` + `accepted_answers`
   - **Layer 2:** AI judge (gpt-4o-mini), temperature 0
5. On API failure → **wrong** (game continues)
6. `games.answer_correct = true` if **any** player correct (drives shared UI)

## 12. AI Integration (Server-Side)

The mobile app **never** holds `OPENAI_API_KEY`. All AI runs on Netlify.

### Question generation

`POST /api/generate-questions` (called from `HomeScreen` on create, `GameScreen` on rematch):

| Input | Notes |
|-------|-------|
| `topic` | Required, ≤200 chars |
| `difficulty` | easy / medium / hard |
| `num_questions` | 3–10 |
| `mc_mode` | boolean |
| `locale` | en, ru, es, ar, fr, de, ja |
| `previous_questions` | Dedup from `question-history.ts` (AsyncStorage on mobile) |

Models via OpenRouter: easy → `gpt-4o-mini`; medium/hard → `gpt-4o` (falls back to mini on error).

Prompts: `lib/quiz-prompts.ts` on web server. MC questions sanitized by `sanitizeMcQuestion()`.

### Session question memory

`question-history.ts` stores up to 24 recent question texts per topic. Passed as `previous_questions` to reduce AI repetition within a session (create + rematch).

### Answer judging philosophy

- Lenient for speech artifacts (filler words, STT errors)
- Strict on factual correctness
- Defaults to wrong on failure so rounds always complete

| Concern | Implementation |

```typescript
// Simplified logic:
if (micMuted) return false;           // user muted
if (mc_mode) return true;           // open mic in MC phases
```

## 21. Network Resilience & Fairness

Layers built into the client:

1. **Realtime + 2.5s polling fallback**
2. **Server clock sync** (30s resync) — prevents deadline desync
3. **Guarded DB updates** — no double scoring or duplicate phase jumps
4. **Initial load retries** — 5× with 1.2s backoff
5. **WebRTC:** 8s presence grace, 3s reconcile loop, ICE restart, AppState/NetInfo recovery
6. **Checking phase timeout** — 15s; if judge client vanishes, round still advances
7. **Transcript throttle** — protects Supabase from speech partial-result floods
8. **Hardcore `answered_at`** — server clock, not local Date, for winner fairness

## 25. Monetization, Free Trial & Paywall (iOS)

This section documents the **creator paywall** added to the native iOS client. It does **not** apply to the web app or to **joining** games — guests can always play for free on any platform.

### Business model summary

| Action | Cost |
|--------|------|
| **Join** a quiz (guest) | Always free |
| **Create** a quiz (host) on iOS | Gated — free trial, then subscription |
| **Create** on web | Not gated by this repo (no paywall in Next.js client) |
| **Play** (answer questions, video, rematch) | Free for everyone once in a game |

Apple requires In-App Purchase for unlocking app functionality on iOS, so paid plans use **StoreKit** via **RevenueCat** (`react-native-purchases`). External payment (e.g. Stripe in-app) is not used for unlocking create access.

### Tiers & allowances

Three logical tiers (`Tier` in `src/lib/purchases.ts`):

| Tier | How obtained | Create allowance | Resets |
|------|--------------|------------------|--------|
| **`free`** | Default; no active subscription | **5 quizzes ever** (`FREE_TRIAL_CREATES`) | Never — lifetime per install |
| **`limited`** | Active `limited` RevenueCat entitlement | **30 quizzes / calendar month** (`LIMITED_MONTHLY_GAMES`) | 1st of each month (local time) |
| **`unlimited`** | Active `unlimited` RevenueCat entitlement | No limit | — |

When the free trial is exhausted → **hard paywall** (cannot create until subscribed or restored). When a `limited` subscriber hits 30 in a month → paywall with `reason: 'monthly'`.

Constants live at the top of `src/lib/purchases.ts`; change entitlement names, trial count, or monthly cap there.

### Subscription products & pricing

Four auto-renewable products in one RevenueCat **Offering**, mapped by package identifier:

| Package ID | Tier | Billing | Fallback price (USD) |
|------------|------|---------|----------------------|
| `LIMITED_MONTHLY` | limited | Monthly | $12.99/mo |
| `LIMITED_ANNUAL` | limited | Annual (~20% off) | $124.99/yr |
| `UNLIMITED_MONTHLY` | unlimited | Monthly | $32.99/mo |
| `UNLIMITED_ANNUAL` | unlimited | Annual (~20% off) | $316.99/yr |

StoreKit shows **localized** prices when offerings load; fallback strings display when billing is unavailable. Suggested App Store Connect product IDs: `ws_limited_monthly`, `ws_limited_annual`, `ws_unlimited_monthly`, `ws_unlimited_annual` (see `SETUP.md` §12).

RevenueCat **entitlements** (must match code):

- `limited` — attaches to both limited monthly/annual products
- `unlimited` — attaches to both unlimited monthly/annual products

If both entitlements are active, **`unlimited` wins** (`tierFromCustomerInfo`).

### Two layers of state: subscription vs usage

Because the app has **no user accounts** (anonymous, link-based games), create usage and subscription status are tracked differently:

| Concern | Storage | Scope | Restorable |
|---------|---------|-------|------------|
| **Subscription tier** | StoreKit / RevenueCat | Apple ID | Yes — **Restore purchases** |
| **Free-trial count** | AsyncStorage `whosmarter-free-creates` | Per install | No |
| **Monthly create count** | AsyncStorage `whosmarter-monthly-creates` (`{ month: "YYYY-MM", count }`) | Per install | No |

Implication: a `limited` subscriber who reinstalls the app keeps their subscription (restore) but **monthly counter resets** on that device. The free trial counter also resets on reinstall (5 more creates) unless/until tied to an Apple ID server-side in a future version.

### Code architecture

```
App.tsx
  initPurchases()                    ← RevenueCat configure (no-op if billing off)
  EntitlementsProvider               ← tier + allowance for whole app
    HomeScreen
      refresh() before create        ← gate before AI spend
      noteCreated() after insert     ← increment quota
      quota pill → Paywall
    PaywallScreen (modal)
      fetchPaywallOptions()          ← StoreKit prices or fallbacks
      purchaseOption() / restore()
      refresh() on success
```

| File | Role |
|------|------|
| `src/lib/purchases.ts` | RevenueCat SDK (lazy-loaded), `Tier`, `fetchTier`, `fetchPaywallOptions`, `purchaseOption`, `restore`, `billingAvailable()` |
| `src/lib/createQuota.ts` | `getCreateAllowance(tier)`, `recordCreate(tier)` — AsyncStorage counters |
| `src/context/EntitlementsProvider.tsx` | Combines tier + allowance; `refresh()`, `noteCreated()`; listens for entitlement updates |
| `src/screens/PaywallScreen.tsx` | Modal UI — plan cards, monthly/annual toggle, subscribe, restore |
| `src/screens/HomeScreen.tsx` | Pre-create gate + quota pill |
| `src/navigation/RootNavigator.tsx` | `Paywall` route, modal presentation |
| `src/lib/config.ts` | `REVENUECAT_IOS_KEY`, `isBillingConfigured()` |

### User-facing paywall flow

1. User taps **Create Challenge** on Home.
2. `handleCreate` calls `refresh()` → `getCreateAllowance(currentTier)`.
3. If `allowed === false`:
   - Navigate to **Paywall** with `reason: 'trial'` (free tier exhausted) or `reason: 'monthly'` (limited cap hit).
   - No API call, no Supabase insert — AI tokens not spent.
4. Paywall shows:
   - Eyebrow: *"You've used all your free quizzes"* or *"You've created all 30 quizzes this month"*
   - Title: **Unlock quiz creation**
   - Subtitle: *Joining quizzes is always free. Choose a plan to keep creating your own.*
   - **Monthly / Annual** toggle (annual tab shows "Save 20%")
   - Two plan cards: **30 games / month** and **Unlimited games** (featured)
   - **Subscribe** → StoreKit sheet via RevenueCat
   - **Restore purchases** (App Store requirement)
   - Footer: auto-renewal terms
5. On successful purchase or restore → alert success → pop back to Home → user can create again.

User can also open Paywall anytime via the **quota pill** ("See plans") without being blocked.

### Graceful degradation (billing not configured)

The app must not crash when IAP is unavailable. Guards throughout `purchases.ts`:

- `billingAvailable()` requires iOS + valid RevenueCat key (not placeholder `REPLACE_WITH`) + native module present
- Lazy `require('react-native-purchases')` — fails silently in Jest / before native rebuild
- If billing off: `fetchTier()` → `'free'`; paywall shows **fallback prices**; **Subscribe disabled**; message *"In-app purchases are not available on this build yet."*
- Users still get the **5-quiz free trial** on every install

This supports local dev, Simulator without StoreKit setup, and Windows-side Metro development before EAS build.

### Enabling production purchases

1. App Store Connect — subscription group + 4 products (see `SETUP.md` §12).
2. RevenueCat dashboard — entitlements `limited` / `unlimited`, default offering with package IDs above, copy public iOS SDK key.
3. Set `EXPO_PUBLIC_REVENUECAT_IOS_KEY=appl_…` in `.env` or `eas.json`.
4. **Rebuild native client** (`expo prebuild --clean`, EAS build, or `expo run:ios`) — `react-native-purchases` is a native module.
5. Test with TestFlight or StoreKit sandbox Apple ID.

### What is not monetized

- Joining as guest (link, QR, Universal Link)
- Playing rounds, scoring, rematch (including host rematch on an **existing** game — rematch does not consume create quota)
- Web client game creation (no paywall in this repo's web counterpart)
- Cross-platform guests in a game created by a paying iOS host

### i18n strings

Paywall copy lives in `src/lib/i18n/messages.ts` (`paywallTitle`, `planLimited`, `planUnlimited`, `billingAnnualSave`, etc.). English and Russian locales include these keys.
